## 貢献の種類

以下のような貢献を歓迎します：

### 1. 誤りの修正

- 誤字・脱字の修正
- 技術的な誤りの訂正
- リンク切れの修正
- コードサンプルの修正

**手順**: Issueを作成するか、直接Pull Requestを送ってください。

### 2. 既存ドキュメントの改善

- 説明の追加・拡充
- 例の追加
- 図表の追加
- 構成の改善
- 読みやすさの向上

**手順**: まずIssueで提案し、同意が得られたらPull Requestを送ってください。

### 3. 新規ドキュメントの作成

- 新しいトピックのドキュメント
- チュートリアル
- ガイド
- リファレンス

**手順**: Issueで提案し、内容について議論した後、Pull Requestを送ってください。

### 4. 翻訳

- 日本語から英語への翻訳
- 英語から日本語への翻訳

**手順**: Issueで翻訳予定を宣言し、Pull Requestを送ってください。

## 始める前に

### 必須の確認事項

1. **既存のIssueを確認**
   - 同じ問題や提案が既に報告されていないか確認してください
   - 重複を避けるため、検索機能を活用してください

2. **最新版を確認**
   - 最新のmainブランチで問題が修正されていないか確認してください
   - ドキュメントが最新の状態であることを確認してください

3. **スコープを明確に**
   - 1つのPull Requestでは1つのトピックに集中してください
   - 複数の異なる変更は、別々のPull Requestに分けてください

## ドキュメントの構造

```
~/Data1/Docs/
├── ja/                    # 日本語ドキュメント
├── en/                    # 英語ドキュメント

    ├── ISSUE_TEMPLATE/    # Issueテンプレート
    └── pull_request_template.md
```

### ディレクトリの使い分け

- **`ja/`**: 完成した日本語ドキュメント
- **`en/`**: 完成した英語ドキュメント
- **`draft/`**: 執筆中や査読待ちのドキュメント

## 貢献の手順

### 1. リポジトリのフォーク

```bash
# GitHubでフォークボタンをクリック
# フォークしたリポジトリをクローン
git clone https://github.com/YOUR_USERNAME/Docs.git
cd Docs
```

### 2. ブランチの作成

```bash
# mainブランチから新しいブランチを作成
git checkout -b fix/typo-in-rust-intro
# または
git checkout -b feature/add-error-handling-guide
```

**ブランチ命名規則**:
- `fix/` - 誤りの修正
- `feature/` - 新規ドキュメント
- `improvement/` - 既存ドキュメントの改善
- `translation/` - 翻訳

### 3. 変更の実施

#### 新規ドキュメントの場合

1. `draft/`ディレクトリで作業開始
2. 完成したら適切な言語ディレクトリに移動

```bash
# 下書き作成
touch draft/新しいドキュメント.md

# 完成したら移動
git mv draft/新しいドキュメント.md ja/技術記事/
```

#### 既存ドキュメントの修正

直接該当ファイルを編集してください。

### 4. コミット

```bash
git add .
git commit -m "docs: Rust入門の誤字を修正"
```

**コミットメッセージの規則**:
- `docs:` - ドキュメントの追加・変更
- `fix:` - 誤りの修正
- `translation:` - 翻訳
- `meta:` - メタデータや設定の変更

### 5. プッシュとPull Request

```bash
git push origin fix/typo-in-rust-intro
```

GitHubでPull Requestを作成してください。

## スタイルガイド

### マークダウン

- 見出しレベルを正しく使用（H1は1つのみ）
- コードブロックには言語を指定
- リストの階層を正しく使用
- 改行とスペースを適切に使用

### 日本語

- です・ます調で統一
- 専門用語は初出時に説明
- 句読点は「、」「。」を使用
- 半角スペースで単語を区切らない（日本語文中）

### 英語

- Use clear, concise language
- Define technical terms on first use
- Use active voice when possible
- Follow standard punctuation rules

### コードサンプル

```rust
// ✅ 良い例：コメントで説明を追加
fn main() {
    let x = 5; // 不変変数
    println!("x = {}", x);
}

// ❌ 悪い例：説明なし
fn main() {
    let x = 5;
    println!("x = {}", x);
}
```

- 動作するコードを記載
- 重要な部分にコメントを追加
- インデントを正しく使用
- エラーになる例は明示的に示す

### リンク

```markdown
<!-- ✅ 相対パスを使用 -->
詳細は[こちら](../basics/intro.md)を参照

<!-- ❌ 絶対パスは避ける -->
詳細は[こちら](/home/user/Docs/ja/basics/intro.md)を参照
```

## レビュープロセス

### Pull Requestの確認事項

レビュアーは以下を確認します：

1. **内容の正確性**
   - 技術的な誤りがないか
   - 情報が最新か
   - コードサンプルが動作するか

2. **品質**
   - 説明が分かりやすいか
   - 適切な例が含まれているか
   - 構成が論理的か

3. **スタイル**
   - スタイルガイドに従っているか
   - マークダウンが正しいか
   - 一貫性があるか

### レビュー後の対応

- コメントへの返信
- 必要な修正の実施
- 修正後の再レビュー依頼

### マージ

レビューが承認され、すべてのチェックがパスしたら、メンテナーがマージします。
